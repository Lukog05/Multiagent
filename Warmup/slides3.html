Exercise 2

BFS via Graph-Search

Graph-search loop

frontier ← {initial}
explored ← ∅
while frontier not empty:
  n ← frontier.pop()
  if goal(n): return plan(n)
  explored.add(n)
  for a in Actions(n):
    n' ← Result(n,a)
    if n' ∉ explored and n' ∉ frontier:
      frontier.add(n')

BFS properties

  • Queue frontier ⇒ expands by increasing depth
  • Complete (finite branching)
  • Optimal (unit step costs)
  • Memory-heavy for large MAPF

Explain the performance differences (what to say)

  • State space size: grows combinatorially with the number of agents and free cells.
  • Branching factor: joint actions scale roughly like k^A (k actions per agent, A agents).
  • Search tree depth: levels needing more coordination require deeper plans → more expansions.
  • Level structure: bottlenecks/corridors increase conflicts → more detours → larger search.

DFS change

Does DFS find solutions faster than BFS? And why?

  • Often faster to first solution: DFS commits to one branch and can reach a goal without exploring siblings.
  • But not guaranteed: DFS can get trapped exploring deep subtrees before hitting a goal.
  • Solution length: typically much longer than BFS (no optimality guarantee).
  • Implementation difference: queue vs stack changes expansion order in the same graph-search loop.

Exercise 4

Best-first search (priority frontier)

Frontier: priority queue ordered by f(n), with a tie-breaker counter.

  • Greedy: f(n) = h(n)
  • A*: f(n) = g(n) + h(n)
  • Weighted A*: f(n) = g(n) + w·h(n)
Exercise 4

Heuristic 1: Goal Count

Definition (MAPF)

h(s) = #{ agents i where pos(i) ≠ goal(i) }

Cheap, but ignores distances and obstacles.

Expected behavior

  • Greedy: prefers states with more agents on goals
  • A*: balances depth (g) with progress (h)
  • Large “plateaus” where many states share same h

Exercise 4

Goal Count: Compare to BFS/DFS

How much faster / better?

  • MAPF01: A* GoalCount expands 2303 ≈ BFS (same scale), but faster runtime due to ordering.
  • MAPF02C: Greedy reduces expansions 101607 → 46301 and time 32.5s → 16.9s.
  • Sliding puzzle: Greedy is dramatically faster (426 expanded) but produces longer plans (46 actions).

Why?

  • GoalCount ignores distance ⇒ many states share the same h-value (plateaus).
  • So A* (g+h) still behaves somewhat like depth/BFS ordering on many instances.
  • Greedy can be faster because it commits to h-improving states, but can sacrifice optimality (longer plans).
  • When the level requires coordination, GoalCount does not reflect the true remaining effort ⇒ timeouts persist.
Exercise 4

Heuristic 2: Distance-sum (preprocessed)

Formal description

Precompute shortest-path distances on the static wall grid, then estimate remaining cost by summing agent→goal distances.

Preprocess:
  dist[c][*] = BFS distances from each free cell c

Heuristic:
  h(s) = Σ_i dist[ pos(agent_i) ][ goal(agent_i) ]
  • More informative than goal-count (captures distance + obstacles)
  • Runtime per state is fast (lookups after preprocessing)
  • Best used with Greedy / (Weighted) A* for scalability
Exercise 4

Distance-Sum Heuristic – Intuition & Trade-offs

Intuition (what it estimates)

  • Instead of only counting misplaced agents, we approximate remaining solution length via shortest-path distances.
  • Obstacle-aware distances act as a lower-bound-ish proxy for “steps still needed”.
  • Moving an agent toward its goal tends to decrease h(s), creating a useful gradient.

Strengths

  • Much fewer expansions (better guidance than GoalCount).
  • Preprocessing makes evaluation cheap per state.
  • Works well with Greedy/Weighted A* on many MAPF levels.

Weaknesses

  • Ignores agent-agent interactions and future conflicts.
  • Underestimates coordination-heavy levels.
  • Can still fail on “reorder”-type instances.

Improvement Analysis (All Benchmarks)

Compare: Uninformed vs GoalCount vs Distance-Sum

  • MAPF01: expansions drop 2303 → 808 with A* when moving from GoalCount to Distance-Sum.
  • MAPF03: GoalCount timed out, but Greedy + Distance-Sum solves (3776 generated).
  • Sliding puzzle: Greedy improves 706 → 153 generated when swapping GoalCount → Distance-Sum.

Why the improvement happens

  • Distance-Sum correlates with true remaining effort (captures obstacles + distance).
  • Fewer plateaus ⇒ fewer wasted expansions.
  • Greedy becomes very strong when h(s) forms a consistent gradient.

What still limits us

  • Some instances (e.g., reorder) remain hard due to coordination constraints.
  • Heuristic ignores multi-agent conflicts and future blocking.
Wrap-up (Ex1–4)

Key takeaways

  • Uninformed search hits exponential blow-up quickly in MAPF.
  • DFS can find a solution where BFS fails under limits, but often at huge plan cost.
  • Best-first + good heuristic shifts the bottleneck: fewer expansions, faster solves.
  • Distance-preprocessing makes h(s) informative and cheap per state.
